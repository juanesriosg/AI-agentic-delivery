# Codex Cloud Branch Spec Trigger

This file describes how Codex Cloud should consume a spec-dispatch task generated by the repository workflow.

## Trigger source

The GitHub workflow `.github/workflows/agent-spec-ingestion.yml` detects a new spec and creates one or both of these:

- a GitHub issue labeled `ai:ready` and `ai:from-spec`,
- a webhook payload to `AGENT_DISPATCH_WEBHOOK_URL`.

## Required Codex Cloud behavior

When Codex Cloud receives a spec task, it must:

1. Clone or open the repository.
2. Checkout the source spec branch from the task payload.
3. Read `AGENTS.md`.
4. Read `.ai/specs/branch-spec-ingestion.yml`.
5. Read `.ai/specs/spec-implementation-pipeline.md`.
6. Read the spec file from the source branch.
7. Run:

```bash
.codex/bootstrap.sh
.ai/scripts/spec-quality-check.py --spec <spec-path> --format markdown
.ai/scripts/generate-test-matrix.py --spec <spec-path>
```

8. Create an implementation branch from the source spec branch:

```bash
git checkout <source-spec-branch>
git checkout -b ai/<spec-id>-<slug>
```

9. Implement, test, self-review, and open a PR back to the source spec branch.
10. Notify the manager using the standard status format.

## Clarifications

Codex Cloud may ask clarifying questions, but must not become idle when safe progress exists.

Examples of safe progress:

- existing behavior characterization,
- test harness setup,
- unit tests for unambiguous logic,
- fixtures,
- component mapping,
- implementation plan,
- bug reproduction.

## Deployment

Codex Cloud must not deploy to production. It may run non-production deployment scripts only when the repository explicitly opts in and all deployment gates pass.
